## Context

当前系统没有账号概念，所有数据（项目、会话、反馈）都是全局共享的。随着功能完善，需要支持多用户隔离，避免不同使用者的资料互相可见。本次变更在 Java 服务（agent-api）中引入轻量级手机号账号体系。

## Goals / Non-Goals

**Goals:**
- 支持手机号注册/登录。
- 用户核心数据（项目、会话消息、反馈）按 `user_id` 隔离。
- 登录态通过 JWT 传递，Java 服务统一鉴权。
- 验证码流程保留短信发送扩展点，但当前固定返回 `8888`，不调用真实短信服务。

**Non-Goals:**
- 不实现密码登录、第三方登录、找回密码等能力。
- 不调用真实短信网关（仅预留接口）。
- 不改造 agent-core 的匿名接口（agent-core 仍由 agent-api 内部调用，不暴露给用户）。

## Decisions

1. **账号体系放在 agent-api**
   - agent-api 是用户请求的入口，天然适合持有登录态和用户信息。
   - agent-core 保持无状态，继续专注于 AI 能力，避免双端都维护用户态。

2. **仅支持手机号作为账号标识**
   - 手机号唯一，无需用户名/邮箱体系，实现简单且符合国内使用习惯。
   - 数据库 `users.phone` 加唯一索引。

3. **验证码固定为 8888**
   - 当前为个人使用，避免接入短信平台产生的费用和复杂度。
   - 保留 `SmsService` 接口及 `MockSmsService` 实现，未来可替换为真实短信实现。

4. **JWT 作为登录态**
   - 使用无状态 JWT，减少服务端会话存储成本。
   - Token 通过 `Authorization: Bearer <token>` 传递，前端在登录后保存。

5. **数据隔离通过 user_id 关联**
   - 现有 `projects`、`session_messages`、`feedbacks` 等表增加 `user_id` 字段。
   - 查询时自动带上当前登录用户的 `user_id` 过滤。

6. **兼容现有匿名数据**
   - 历史数据的 `user_id` 设置为 `NULL` 或一个默认系统用户，避免强制迁移。
   - 登录后新产生的数据绑定到真实用户。

## Risks / Trade-offs

- **[Risk] 匿名数据归属不清** → 历史数据保留为系统用户所有，新数据按账号隔离，不影响现有记录。
- **[Risk] 固定验证码存在安全风险** → 当前仅个人使用，后续接入真实短信时替换 `MockSmsService` 即可。
- **[Trade-off] 前端需要改造登录流程** → 所有需要用户身份的操作都要先登录，首次使用有交互成本。

## Migration Plan

1. 新增 `users` 表。
2. 给 `projects`、`session_messages`、`feedbacks` 等表增加 `user_id` 字段。
3. 部署后，历史数据默认 `user_id = 1`（系统用户）。
4. 前端新增登录页，登录后所有请求携带 JWT。
