## Context

The current intent recognition engine in `agent-core/app/services/intent_recognition.py` uses a pipeline of exact, alias, fuzzy, and semantic matching against a fixed taxonomy, with an LLM fallback only for missing fields. This works well for closed vocabularies (space types, display points, styles, materials) but fails for open-ended concepts such as themes, budgets, colors, brands, and timelines. The result is partial intent understanding and user frustration.

This design introduces an LLM-first structured-output extractor backed by a small post-validation layer. The LLM is responsible for understanding natural language; the validator is responsible for normalizing values, constraining required fields, and deciding when to ask for clarification.

## Goals / Non-Goals

**Goals:**
- Recognize both closed-set and open-ended intent fields from a single user utterance.
- Normalize budget to an integer RMB value and map common aliases to canonical taxonomy values.
- Provide per-field confidence and source metadata.
- Trigger clarification when required fields are missing or low-confidence.
- Capture user corrections to intent fields for future learning.

**Non-Goals:**
- Building the admin UI for reviewing corrections (Change 3).
- Automatic alias expansion from corrections (Change 2).
- Multi-turn dialogue management beyond a single clarification prompt.
- Changing the image generation pipeline.

## Decisions

### 1. LLM structured output as the primary extractor
**Decision:** Use a pydantic `IntentOutput` schema and a system prompt that asks the LLM to return JSON matching the schema.
**Rationale:** The LLM can generalize across phrasing ("30万", "300k", "预算三十万左右") and can infer open fields ("圣诞节" → theme) without hand-coded rules.
**Alternative considered:** Keep the rule pipeline and add more regexes. Rejected because the rule set would grow indefinitely and still miss synonyms.

### 2. Rule-based post-validation instead of pre-filtering
**Decision:** After LLM extraction, run a `Validator` that normalizes budget, maps aliases, checks required fields, and computes confidence.
**Rationale:** This keeps the LLM prompt simple and gives us deterministic control over value constraints and fallback behavior.
**Alternative considered:** Put all constraints in the LLM prompt. Rejected because prompt-based constraints are harder to test and change.

### 3. Open taxonomy with canonical mapping
**Decision:** The taxonomy YAML remains a reference/canonical source, but the LLM may output values not in the taxonomy. The validator maps known aliases to canonical values and preserves unknown values as-is.
**Rationale:** Prevents the system from silently dropping valid user input while still maintaining standard values where possible.

### 4. Confidence and clarification thresholds
**Decision:** Each field carries a confidence score and source (`llm`, `validated`, `default`). Required fields below a configurable threshold trigger `needs_clarification` with a generated follow-up question.
**Rationale:** Makes the system behave more like an agent that asks instead of guesses.

### 5. Capture corrections in the existing feedbacks table
**Decision:** Extend `feedbacks` with `category`, `intent_field`, `original_value`, `corrected_value`, `processed`, and `notes` rather than creating a separate table.
**Rationale:** Reuses the existing feedback endpoint and keeps the schema simple. A dedicated `feedback_corrections` table can be added in Change 2 if the learning loop needs it.

## Risks / Trade-offs

| Risk | Mitigation |
|---|---|
| LLM returns malformed JSON or hallucinates fields | Use pydantic validation with retries; on failure fall back to rule-based extraction and mark confidence low. |
| LLM output drifts after prompt changes | Maintain golden regression tests that run on every change. |
| Slower than pure rule matching | The LLM call is already part of the pipeline; structured output adds negligible overhead compared to the current free-text fallback. |
| Sensitive user input in LLM prompts | No PII handling change; same data was already sent to the LLM in the fallback path. |

## Migration Plan

1. Deploy new `intent_recognition.py` alongside old code under a feature flag or local test.
2. Run golden tests and compare old/new outputs.
3. Update `agent-api` Flyway migration to add correction columns to `feedbacks`.
4. Update FeedbackController/Service to accept new fields.
5. Switch input_parser to use the new extractor.
6. Rollback: revert to previous commit; Flyway columns are nullable or have defaults so rollback is safe.

## Open Questions

- Should the clarification question be generated by the LLM or use static templates per missing field?
- Should unknown `space_type` values be accepted or always trigger clarification?
