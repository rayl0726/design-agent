## Context

当前 `agent-core` 的图像生成流程将用户需求简单拼接为 prompt 后调用外部文生图服务。这种粗放方式导致：
- 主体不突出，如“中庭”场景没有强调中间区域；
- 背景喧宾夺主，商业展示元素被环境细节淹没；
- 缺乏对负向提示的系统性控制；
- 用户对生成结果的不满无法回流到识别和生成环节。

本变更引入结构化 prompt 模板和反馈驱动的学习飞轮，解决上述问题。

## Goals / Non-Goals

**Goals:**
- 建立主体-环境-视角-光照-风格-负向提示的 prompt 结构，突出主体、弱化背景。
- 为购物中心中庭提供默认模板和可配置负向提示。
- 将模板渲染接入现有图像生成链路，保持下游接口不变。
- 基于 `feedbacks` 表构建学习飞轮：记录纠错、扩展别名、积累 few-shot 示例、管理 prompt 版本。

**Non-Goals:**
- 不替换现有图像生成模型，只优化输入 prompt。
- 不自动重新训练模型，学习以规则化 alias/few-shot 扩展为主。
- 不在本变更中实现完整的 Admin 后台 UI（属于 Change 3）。

## Decisions

- **Prompt 模板使用 YAML/JSON 配置而非硬编码**：便于运营人员后期调整，支持版本管理。
- **模板渲染采用 Jinja2**：与现有 `agent-core/app/templates` 技术栈一致，轻量且可扩展。
- **负向提示按空间类型和主题分类**：默认提供“通用背景弱化”、“人物过多”、“低质量”等选项，用户可在请求层覆盖。
- **学习飞轮复用 `feedbacks` 表**：避免新增表，利用 Change 1 已扩展的 `category`、`intent_field`、`original_value`、`corrected_value` 字段。
- **Few-shot 示例存储为本地 JSONL**：按空间类型和主题索引，生成前注入到 LLM prompt 中。

## Risks / Trade-offs

- [Prompt 模板过度复杂] → 初始只实现一个通用结构 + 一个购物中心中庭模板，验证后再扩展。
- [负向提示过强导致图像失真] → 默认负向提示保守，允许用户关闭或覆盖。
- [学习数据不足时 alias 扩展 noisy] → 同一纠错出现 3 次后才自动扩展，并支持人工审核标记。
- [Prompt 版本切换影响历史项目] → 每个项目记录使用的 prompt 版本，历史项目不随版本升级而改变。

## Migration Plan

1. 新增模板和配置目录，旧项目不受影响。
2. 图像生成服务新增 `PromptTemplateRenderer`，默认对无模板的项目使用 legacy 拼接逻辑。
3. 部署后通过 A/B 生成对比验证模板效果，再切换为默认使用。

## Open Questions

- 是否需要为每个 space_type 维护独立模板，还是先以“中庭”和“快闪店”两个高频场景为主？
- 负向提示的默认值是否按预算等级区分（如低预算更强调简洁背景）？
