## Context

通用 Agent 的联网搜索当前只依赖 Bing，返回标题和摘要，缺少对网页正文的综合总结；用户也无法感知搜索/总结阶段，容易误判为卡顿。本次变更把搜索扩展为 Bing + Baidu 双源，抓取正文后用 LLM 生成摘要，并通过 SSE 状态事件实时反馈进度。

## Goals / Non-Goals

**Goals:**
- `WebSearchTool` 同时搜索 Bing 与 Baidu，合并去重后返回 Top N 结果。
- 对 Top 3 条结果抓取网页正文；失败时 fallback 到搜索摘要。
- 使用 LLM 对正文/摘要生成一段中文综合摘要，并保留来源链接。
- 百度结果过滤广告条目。
- 前端工具卡片展示“搜索中... / 总结中... / 已完成”。

**Non-Goals:**
- 不实现通用网页浏览代理（只抓取搜索结果页）。
- 不改写 LLM 的调用决策逻辑（仍由 LLM 决定何时调用 `web_search`）。
- 不替换现有美陈 Agent 的搜索能力。

## Decisions

1. **搜索并发：Bing 与 Baidu 并行**
   - 理由：减少整体延迟；任一源失败仍可继续。
   - 替代方案：串行搜索，延迟高但实现简单；并行更优。

2. **去重策略：URL 归一化 + 标题 Jaccard 相似度**
   - URL 去掉 scheme、`www`、尾部斜杠和查询参数后比较；再对标题做 Jaccard > 0.6 的去重。
   - 理由：实现轻量，不依赖外部库。

3. **正文提取：优先 BeautifulSoup 提取主内容区，超时 fallback 到摘要**
   - 优先选择 `<article>`、`<main>`、`role="main"`、class/id 含 `content`/`article`/`post` 的节点；若失败则使用搜索摘要。
   - 理由：无需引入重量级解析库；满足总结需求。

4. **摘要生成：单轮 LLM 调用，输入拼接 Top 3 正文/摘要**
   - 系统提示词要求用中文回答用户问题，保留来源链接。
   - 理由：减少调用次数，降低延迟。

5. **状态事件：新增 `tool_progress` SSE 事件**
   - `tool_start` 插入卡片，状态“搜索中...”；`tool_progress(status=summarizing)` 更新为“总结中...”；`tool_result` 更新为“已完成”。
   - 替代方案：在 `tool_result` 中分多次推送，无法区分中间状态；`tool_progress` 语义更清晰。

6. **广告过滤：标签/域名/文本三层过滤**
   - 过滤带 `data-tuiguang`、class 含 `ec_` 或 `ad-` 的节点；黑名单域名（`e.baidu.com` 等）；标题/摘要含“广告”“推广”的条目降级或剔除。
   - 理由：多层过滤可提高准确率，且实现简单。

## Risks / Trade-offs

- **Baidu 页面结构变化导致解析失效** → 增加单元测试覆盖常见结果页；失败时回退到 Bing 结果。
- **正文抓取超时拖慢整体响应** → 抓取阶段使用 `asyncio.Semaphore(max=3)` 与 10s 超时；超时时用摘要替代。
- **LLM 摘要成本高/慢** → 摘要输入限制在 3000 token 以内；摘要失败返回原始结果列表。
- **百度反爬** → 使用常见 User-Agent 并控制请求频率；若被拦截则只使用 Bing。
- **广告过滤误删有效结果** → 优先过滤明确广告标记；误删后仍保留至少 3 条非广告结果兜底。
